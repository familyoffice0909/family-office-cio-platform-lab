# R1.3.1.1 Registry Authority Architecture

- **Owner:** Chief Architect / Platform Governance
- **Status:** Implemented; review, merge, and release closure pending
- **Wave:** R1.3.1.1 — Registry Authority
- **Decision:** [ADR-004 — Registry Authority](../adr/ADR-004-REGISTRY-AUTHORITY.md)

## Objective and boundary

Registry Authority governs the identity, ownership, version, authority,
validation function, and registration status of platform registries. It does
not replace a registry, copy its business data, or intercept existing consumer
access.

This wave adds only registration, discovery, validation, initial adoption,
tests, and governance evidence. Writer, Reporting, Failure, and Certification
Authorities; RunContext; dependency enforcement; runtime redesign; spreadsheet
redesign; and business-logic changes remain outside scope.

## Component contract

```text
Existing Registry Business Data
  ├── FO_SHEETS
  ├── FO_REQUIRED_DASHBOARD_SHEETS
  └── FO_MARKET_SYMBOLS (Market Symbol Registry)
             │ references supplied only to validators
             ▼
Registry Authority
  ├── registration API (fail closed)
  ├── metadata-only discovery API
  └── validation engine
             │
             ▼
Immutable registration metadata views
```

The authority catalog is lazy, in-memory, and deterministic. Source code in
GitHub remains authoritative for registrations. The catalog performs no
workbook or Drive reads or writes.

## Metadata model

| Field | Type | Required | Rule |
|---|---|---|---|
| `registryName` | String | Yes | Trimmed, non-empty, case-insensitively unique |
| `owner` | String | Yes | Trimmed, non-empty, case-insensitively unique |
| `version` | String | Yes | Trimmed and non-empty |
| `authority` | String | Yes | Trimmed and non-empty |
| `validationFunction` | Function | Yes | Callable; must return `true`, `PASS`, or `PASS WITH OBSERVATIONS` |
| `registrationStatus` | String | Yes | `REGISTERED`, `SUSPENDED`, or `RETIRED` |

The internal registration record also holds a reference to the registry-owned
business data solely for its validation function. Discovery omits that
reference, so Registry Authority cannot become an alternate business-data API.

## API behavior

### Registration

`foRegisterRegistry(metadata)` normalizes the candidate, validates the entire
candidate catalog, and appends an immutable record only when every control
passes. A rejected registration does not mutate the catalog and throws an error
that includes the validation code and description.

### Discovery

`foDiscoverRegistries(criteria)` returns all immutable metadata views or filters
case-insensitively by registry name, owner, version, authority, or registration
status. A string criterion is interpreted as the exact registry name.

`foDiscoverRegistry(registryName)` returns the exact metadata view or `null`.
Neither API returns registry business data.

### Validation

`foValidateRegistryAuthority()` validates all adopted registrations and returns
a structured, immutable result with authority name, overall status, counts, and
errors. Each error includes a stable code, registration index, and message.

| Code | Detection |
|---|---|
| `DUPLICATE_REGISTRY_NAME` | A normalized registry name already exists |
| `DUPLICATE_OWNERSHIP` | A normalized accountable owner already exists |
| `MISSING_METADATA` | Name, owner, version, or authority is blank |
| `MISSING_VALIDATION_FUNCTION` | Validation function is absent or not callable |
| `INVALID_REGISTRATION_STATUS` | Status is outside the governed status set |
| `REGISTRY_VALIDATION_FAILED` | Registry-owned validation returns failure or throws |

`foRunRegistryAuthoritySmokeTest()` requires a passing catalog containing
exactly the three approved registrations. It has no spreadsheet side effects.

## Initial adoption

| Registry name | Owner | Version | Authority | Status | Business-data owner remains |
|---|---|---|---|---|---|
| `FO_SHEETS` | Platform Configuration | `1.0.0` | Registry Authority | `REGISTERED` | `FO_SHEETS` |
| `FO_REQUIRED_DASHBOARD_SHEETS` | Platform Health | `1.0.0` | Registry Authority | `REGISTERED` | `FO_REQUIRED_DASHBOARD_SHEETS` |
| Market Symbol Registry | Market Data | `1.0.0` | Registry Authority | `REGISTERED` | `FO_MARKET_SYMBOLS` |

No other registry is adopted or created by this wave.

## Compatibility and operational behavior

- Existing registry constants, functions, callers, and access patterns remain
  unchanged.
- No orchestrator, module order, menu, trigger, schedule, lock, manifest,
  authorization scope, worksheet schema, or spreadsheet service changes.
- No portfolio, recommendation, reporting, certification, or other business
  logic changes.
- Bootstrap is idempotent within an execution. Registration is atomic relative
  to validation because mutation occurs only after all controls pass.
- Source rollback requires a Git revert only; there is no persistent authority
  data or workbook state to restore.

## Validation and evidence

Automated tests cover adoption, discovery, immutable metadata views, all
required error categories, registry-owned validation failure, failed
registration rollback, and the smoke-test entry point. The complete validation
record is in [R1.3.1.1 Registry Authority Evidence](../validation/R1.3.1.1-REGISTRY-AUTHORITY-EVIDENCE.md).
